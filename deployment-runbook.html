1 Containerize & manual first deploy

Dockerfile

Multi-stage build, Node 22 Alpine. NODE_ENV=production is set only in the final runner stage — setting it earlier would make pnpm install skip devDependencies that next build needs (typescript, tailwindcss, etc.).

Dockerfilerepo root · implemented
# syntax=docker/dockerfile:1

# ---- Base -------------------------------------------------------------------
FROM node:22-alpine AS base

# libc6-compat is required by some native Node addons on Alpine (e.g. sharp);
# python3/make/g++ are a safety net in case any dependency needs to build from
# source rather than use a prebuilt binary.
RUN apk add --no-cache libc6-compat python3 make g++

# Pin the exact pnpm major used to generate pnpm-lock.yaml (lockfileVersion 9.0)
RUN corepack enable && corepack prepare pnpm@9.15.0 --activate

WORKDIR /app

# ---- Dependencies (cached separately from source changes) -------------------
FROM base AS deps
COPY package.json pnpm-lock.yaml .npmrc pnpm-workspace.yaml ./
RUN pnpm install --frozen-lockfile

# ---- Build --------------------------------------------------------------------
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .

ENV NEXT_TELEMETRY_DISABLED=1
RUN pnpm build

# drizzle-orm/postgres-js/migrator is never imported anywhere in the app
# itself (only in our own migrate script below), so Next's standalone output
# tracing drops that submodule. Materialize the two pure-JS, zero-dependency
# packages the migration entrypoint needs, dereferencing pnpm's symlinks so
# they survive being copied into the slim runtime stage on their own.
RUN mkdir -p /app/runtime-only/node_modules/postgres /app/runtime-only/node_modules/drizzle-orm \
  && cp -rL /app/node_modules/postgres/. /app/runtime-only/node_modules/postgres/ \
  && cp -rL /app/node_modules/drizzle-orm/. /app/runtime-only/node_modules/drizzle-orm/

# ---- Runtime (lean, production-only) -----------------------------------------
FROM node:22-alpine AS runner
WORKDIR /app

ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
ENV HOSTNAME=0.0.0.0
ENV PORT=3000

RUN addgroup --system --gid 1001 nodejs \
  && adduser --system --uid 1001 nextjs

# Next.js standalone server + its minimal traced node_modules
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
# No public/ directory exists yet. If one is added later, uncomment:
# COPY --from=builder /app/public ./public

# Drizzle migration assets: SQL files + our small runtime migration script
COPY --from=builder /app/drizzle/migrations ./drizzle/migrations
COPY --from=builder /app/drizzle/migrate.mjs ./drizzle/migrate.mjs

# Explicit prod-only packages needed by drizzle/migrate.mjs (see builder stage)
COPY --from=builder /app/runtime-only/node_modules/postgres ./node_modules/postgres
COPY --from=builder /app/runtime-only/node_modules/drizzle-orm ./node_modules/drizzle-orm

COPY docker-entrypoint.sh /usr/local/bin/docker-entrypoint.sh
RUN chmod +x /usr/local/bin/docker-entrypoint.sh

USER nextjs
EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/api/health || exit 1

ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["node", "server.js"]
Real issue hit & fixed during verification pnpm-workspace.yaml had no packages: field. Locally-installed pnpm 11 tolerates this, but pnpm 9.15.0 (pinned in the Dockerfile via corepack) fails with ERROR packages field missing or empty. Fixed by adding: 
packages:
  - '.'
to the top of pnpm-workspace.yaml. Confirmed via an isolated repro before touching the real file.

Migration runner (no drizzle-kit needed at runtime)

Standard Drizzle-recommended pattern: drizzle-orm/postgres-js/migrator ships inside the already-production drizzle-orm dependency, so no dev tooling needs to be bundled into the runtime image.

drizzle/migrate.mjsimplemented
import postgres from 'postgres';
import { drizzle } from 'drizzle-orm/postgres-js';
import { migrate } from 'drizzle-orm/postgres-js/migrator';

async function run() {
  const connectionString = process.env.DATABASE_URL;
  if (!connectionString) {
    console.error('DATABASE_URL is not set — cannot run migrations.');
    process.exit(1);
  }

  const sql = postgres(connectionString, { max: 1 });
  const db = drizzle(sql);

  console.log('Running Drizzle migrations...');
  await migrate(db, { migrationsFolder: './drizzle/migrations' });
  console.log('Migrations complete.');

  await sql.end();
}

run().catch((err) => {
  console.error('Migration failed:', err);
  process.exit(1);
});

✋ VPS Runbook — manual, hand-executed

A. Create an isolated database + role for hub

SSH + psqlshell
ssh mykola@<vps-host>
cat /home/mykola/infra/db/.env        # note the existing POSTGRES_USER (superuser) + POSTGRES_PASSWORD
cd /home/mykola/infra/db
docker compose exec postgres psql -U "$POSTGRES_USER" -d postgres

Inside psql (generate the password beforehand with openssl rand -base64 24):

SQLrun inside psql
CREATE ROLE hub WITH LOGIN PASSWORD 'REPLACE_WITH_A_STRONG_RANDOM_PASSWORD';
CREATE DATABASE hub OWNER hub;
\c hub
GRANT ALL PRIVILEGES ON SCHEMA public TO hub;
\q
Result A dedicated database + role, isolated from anything else already in that Postgres instance. Connection string for later steps: postgresql://hub:<password>@postgres:5432/hub (postgres resolves via Docker DNS once the hub container joins db_network).

B. Coolify app setup — Docker Compose build pack

Decision Use Coolify's Docker Compose resource type, not "Dockerfile". The Dockerfile build pack auto-generates its own compose file with no way to attach an extra external network. Docker Compose lets you supply the file above as-is — Coolify injects Traefik labels + its internal network for routing, but preserves the db_network: external: true block.
  1. Coolify dashboard → your Server/Project → + New Resource.
  2. Source: your Git repository (my-projects), branch main. Add a deploy key or GitHub App connection under Coolify's Sources settings first if it doesn't already have access.
  3. Build Pack: Docker Compose, pointed at docker-compose.yml at the repo root.
  4. General settings → Ports Exposes: 3000 (matches the Dockerfile's EXPOSE 3000).
  5. Domains: https://blonskyi.dev. Coolify auto-generates the Traefik router and requests a Let's Encrypt certificate once DNS resolves to the VPS.
  6. Environment Variables tab: add every var from the table below.
  7. Click Deploy. Coolify clones the repo, runs docker compose build (using the Dockerfile) then docker compose up -d.
  8. Verify network attachment over SSH:
    verifyshell
    docker ps --filter "name=hub"
docker network inspect db_network --format '{{json .Containers}}' | python3 -m json.tool
    If the hub container is missing from that list, attach it manually as an immediate fix: docker network connect db_network <actual-container-name>, then check Coolify's raw compose preview to see why the declared network didn't survive, and re-verify after the next redeploy.
DNS / Cloudflare caveat — verify, don't assume The original docs mention Cloudflare orange-cloud proxy with SSL mode Full. Traefik's HTTP-01 challenge generally works through Cloudflare's proxy as long as SSL mode stays "Full" (not "Flexible") and port 80 reaches Traefik. If cert issuance fails on first deploy, temporarily grey-cloud the DNS record for blonskyi.dev, let Traefik issue the cert, then re-enable the proxy.

D. Apply migrations + run the seed once

The entrypoint already runs drizzle/migrate.mjs automatically before node server.js on every container start/redeploy — nothing manual needed there. The seed (sqlpanel project row) is deliberately not automatic; run it once from your local machine over an SSH tunnel, so you can use your existing pnpm/tsx tooling without needing dev tooling inside the container:

one-time seedshell, from your machine

.github/workflows/deploy.yml

Strict linear gate via needs: — a failure anywhere stops everything downstream; nothing gets pushed or deployed unless lint, typecheck, integration tests, and e2e tests all pass first.

name: Deploy

on:
  push:
    branches: [main]

concurrency:
  group: deploy-main
  cancel-in-progress: false

jobs:
  lint:

      - uses: pnpm/action-setup@v4

          version: 9.15.0

        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm test   # pretest/posttest already manage docker-compose.test.yml

  e2e-tests:
    runs-on: ubuntu-latest
    needs: [integration-tests]
    env:
      DATABASE_URL: postgres://ci:ci@localhost:5432/ci_dummy
      AUTH_SECRET: ci-dummy-secret-not-for-prod-use
      AUTH_TRUST_HOST: "true"
      GOOGLE_CLIENT_ID: ci-dummy-client-id
      GOOGLE_CLIENT_SECRET: ci-dummy-client-secret
      OWNER_EMAIL: ci@example.com
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9.15.0
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec playwright install --with-deps chromium
      - run: pnpm test:e2e

  build-and-push:
    runs-on: ubuntu-latest
    needs: [e2e-tests]
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3

          push: true

    needs: [build-and-push]

Why the built-in GITHUB_TOKEN is enough No separate PAT is needed — the image lives at ghcr.io/mykola-blonskyi/my-projects, the same org/repo the workflow runs in, and permissions: packages: write on the job grants the built-in token push rights to that package.

Coolify webhook wiring — still your manual step

Generating it: open the hub application in Coolify (the one created manually in Phase 1) → its Webhooks section exposes a deploy URL of the form https://<your-coolify-domain>/api/v1/deploy?uuid=<application-uuid>&force=false. Authentication is a Bearer token generated separately under your Coolify user profile's API Tokens page. Together these become two GitHub repo secrets:

  • COOLIFY_WEBHOOK_URL → the full URL above
  • COOLIFY_WEBHOOK_TOKEN → the API token

Add both under GitHub repo → Settings → Secrets and variables → Actions → New repository secret.

Switching from build-from-git to pull-from-GHCR Drop the build: key from the hub service in docker-compose.yml, keeping only image: — Coolify's compose deployment is just docker compose up -d under the hood, so a service with no build: key is simply pulled rather than built. No change to Coolify's resource type is required beyond this one-line diff.

Prerequisite before flipping the switch ghcr.io/mykola-blonskyi/my-projects is very likely a private package. Coolify's Docker daemon needs registry credentials to pull it — configure a GHCR read-only PAT (read:packages scope) as a registry credential in Coolify (Server → Docker Registries, or per-application registry settings), or docker login ghcr.io on the VPS host directly. Do this once before flipping the compose file, or the first pull-based redeploy fails with an image-pull auth error.

Phase 1 → Phase 2 handoff

Before Phase 2 exists: pushing to main does nothing automatically — deploys happen only when you manually click "Redeploy" in Coolify (re-clones and rebuilds from git).

After Phase 2 is live: every push to main automatically runs the full lint → typecheck → integration → e2e gate; only if all four pass does it build the image, push to GHCR, and call the Coolify webhook, which pulls that new image and recreates the container — no manual Coolify UI interaction needed for routine deploys. The UI remains available as a fallback: with only a :latest tag configured, the fastest rollback for a bad deploy that somehow got through is git revert the offending commit on main and let the pipeline redeploy the reverted state (adding SHA-tagged images to the workflow would enable a faster image-level rollback, worth doing before relying on it).
